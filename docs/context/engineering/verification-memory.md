## Known verification rules

- `git status --short --branch` is required for every task.
- Docs-only tasks require diff review and secret/private endpoint scan by Security reviewer.
- Python automation tasks should run `python -m pytest` if test framework exists.
- Android runtime tasks require approved device/APK/config.
- Missing runtime dependencies must produce `blocked` report, not fake pass.
- Exported component guard skeletons must not publish raw component inventories or executable runtime/device recipes.
- Release gate generators must fail closed; R0/R1 gates require `status=pass` and `evidence_status=confirmed`.
- Manual runtime screen/focus map generators must fail closed; all prerequisite gates require `present=true` and `evidence_status=confirmed`, while template observations remain `not_run`/`unknown`.
- Fixture contracts must fail closed; future fixture use requires confirmed owner approval, bounded scope, synthetic-only data, resource budget, redaction, ignored evidence storage and cleanup/rollback.
- Compatibility/device matrix generators must fail closed; future compatibility execution requires confirmed build, target class, config, fixture, redaction, ignored evidence storage, cleanup/rollback and reviewer approvals.
- WebView/payment safe generators must fail closed; future execution requires confirmed build, target, config, WebView fixture policy, staging-only non-real-payment policy, synthetic user policy, resource budget, redaction, ignored evidence storage, cleanup/rollback and reviewer approvals.
- CI/nightly smoke generators must fail closed; future live CI scheduling or runtime lanes require confirmed static scope, schedule policy, repository safety policy, resource budget, redaction, ignored evidence storage, artifact retention, dependency policy and reviewer approvals.
- Navigation transition map generators must fail closed; future runtime transition execution requires confirmed build, target, config, transition scope, input event policy, fixture policy, resource budget, redaction, ignored evidence storage, cleanup/rollback and reviewer approvals.
- Safe task prioritization maps do not approve execution; future runtime/device/APK/WebView/WebRTC/payment/network/live CI tasks remain blocked until dependency approvals are recorded as present, confirmed and independently reviewed.
- Safe backlog refresh docs do not approve execution; proposed follow-up tasks remain public-safe docs/static/fail-closed work unless a future task records confirmed approvals and independent review.
- Approval metadata validation does not approve runtime pass; future TASK-005 may start only after metadata is `approved_for_limited_runtime`, required reviews are approved/confirmed and a separate runtime task runs under confirmed approvals.
- Release gate pass decisions require both passing/confirmed R0/R1 gates and required reviewer approvals/confirmations.
- Exported component guard prerequisites require both `present=true` and `evidence_status=confirmed`.
